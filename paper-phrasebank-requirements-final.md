# paper-phrasebank 需求文档（终版）

> **项目名称**：`paper-phrasebank`（CLI 命令：`ppb`）
>
> **项目定位**：个人学术写作辅助工具。把论文喂进去，自动抽取出适合复用在学术写作里的模板句，经人工确认后存入本地向量库；写论文时用自然语言描述需求，检索出可直接参考或改写的句子。
>
> **核心场景**：读论文时顺手存好句子，写论文时不用凭记忆翻旧论文，直接搜。

---

## 1. 名词约定

| 术语 | 定义 |
|------|------|
| **候选句子** | LLM 从论文中抽取、尚未经人工确认的句子 |
| **功能分类** | 按写作功能给句子分类（共六类，见 4.3 节） |
| **建库链路** | 论文输入 → 句子写入向量库的完整流程，`ppb extract` 一个命令走完 |
| **检索链路** | 用户描述使用场景 → 返回匹配句子的流程 |
| **查询理解层** | 将自然语言描述转换为结构化检索参数的轻量模块（CLI 入口专用） |
| **核心检索层** | 接收结构化参数、执行编码与向量检索的模块（CLI 与未来 MCP 接口共用） |

---

## 2. 整体架构

系统分为两条独立链路，共用同一套 Embedding 模型与向量库，除此之外互不依赖，可分开开发与测试。

```
建库链路：
论文文件 → 格式识别与解析 → 脏数据过滤 → 多级文本分块
        → LLM 解耦抽取（元数据 + 候选句）→ 待审核队列（支持中断恢复）
        → 交互式人工审核 → 本地向量化编码 → 写入 Chroma

检索链路：
自然语言查询 → 本地向量化编码 → Chroma 相似度检索 → 终端格式化输出
```

## 3. 功能模块详细设计

### 3.1 配置管理（交互式，全程不碰文件）

**核心原则**：所有配置项都通过 `ppb config` 命令在终端交互式完成，用户**不需要也不应该**手动编辑任何配置文件。`.env` 只是持久化的载体，由程序自动读写，用户无需感知其路径与格式。

#### 3.1.1 首次运行引导

首次执行任意 `ppb` 命令且未检测到配置文件时，自动进入引导式配置流，按顺序完成以下设置：

**第一步：LLM 提供商选择**
- 终端方向键菜单：
  - `DeepSeek（官方 API）`
  - `OpenAI Compatible（通义千问 / Kimi / Ollama 等兼容接口）`
- **DeepSeek 分支**：自动绑定 `base_url` + `model_name`，仅要求输入 `API Key`（密码掩码输入，不回显）。
- **OpenAI Compatible 分支**：依次要求输入 `Base URL`、`API Key`（支持回车跳过，适配本地免密模型，如 Ollama）和 `Model Name`。

**第二步：OCR 后端配置**
- 终端方向键菜单：
  - `暂不启用 OCR（图片型 PDF 页面将被跳过）`
  - `MinerU API`
  - `PaddleOCR API`
- 选择任一 OCR 后端后，交互式输入对应的 `API Key` 和 `Base URL`（如适用）。

**第三步：确认与写入**
- 汇总展示所有已填配置项及脱敏后的密钥（如 `sk-****a1b2`），请用户确认。
- 确认后写入配置文件，无缝恢复执行被拦截的原始命令。

> 引导流程中每一步都支持 `Ctrl+C` 安全退出，已填写的内容不保存。

#### 3.1.2 日常配置管理

提供 `ppb config` 命令用于随时查看和修改配置，全程交互式，不走文件：

```
ppb config                   交互式菜单，展示当前所有配置项及脱敏值，
                              支持方向键选择项目后修改

ppb config show               只读展示当前完整配置（密钥脱敏）

ppb config set <key> <value>  快速设置单项（高级用户快捷方式，非主推用法）
```

配置项清单（全部支持交互式修改）：

| 配置项 | 说明 | 交互式输入方式 |
|--------|------|---------------|
| LLM 提供商 | DeepSeek / OpenAI Compatible | 方向键选择 |
| LLM Base URL | API 地址 | 文本输入（切换提供商时自动填充默认值） |
| LLM API Key | 密钥 | 密码掩码输入 |
| LLM Model Name | 模型名 | 文本输入（自动填充默认值） |
| OCR 后端 | 关闭 / MinerU / PaddleOCR | 方向键选择 |
| OCR API Key | 密钥 | 密码掩码输入 |
| OCR Base URL | API 地址 | 文本输入（自动填充默认值） |

#### 3.1.3 配置缺失时的兜底行为

- 若 LLM 未配置，`ppb extract` 在执行前自动跳转到配置引导流程，而不是报错退出。
- 若 OCR 未配置，`ppb extract` 遇到图片型页面时打印一行警告并跳过该页，不中断整体流程。

### 3.2 输入解析与预处理

- **格式支持**：PDF、DOCX，根据文件后缀自动选择解析分支，产物统一为"文本块列表"，后续模块不感知原始格式。

**PDF 分支**（PyMuPDF）

- 逐页提取文本层。
- 判断图片型页面：该页提取文本字符数低于阈值（初始 50 字符，实测后调整）且页面包含图像对象时，判定为图片型页面。
- 图片型页面调用 OCR 接口（MinerU API / PaddleOCR API，可插拔，通过配置项切换）。
- 同一篇论文内允许文本提取与 OCR 混合使用，按页处理，不要求整篇统一。

**DOCX 分支**（python-docx）

- 读取正文；若使用标题样式（Heading 1/2 等），按标题切分章节；否则退化为段落级分块，与 PDF 分支兜底策略一致。

**脏数据过滤**

- 在解析后、分块前，通过正则清洗 PDF 的页眉、页脚、孤立页码以及引用上标（如 `[12, 13]`），防止噪音干扰句子边界判定。

### 3.3 多级降级文本分块

避免固定长度硬截断破坏上下文，采用三级降级策略：

1. **一级切分（语义章节）**：DOCX 依赖 Heading 样式；PDF 依赖字体/粗体跳变，或正则匹配常见学术标题（如 `^1\. Introduction`、`Method`、`Results`、`Discussion` 等关键词）。
2. **二级切分（自然段落）**：章节内优先按双换行符 `\n\n` 或段落缩进划分为段落块。
3. **三级切分（标点回退兜底）**：若自然段落超过 `max_chunk_size`（如图表长说明），向后回退寻找最近的句号、问号或分号（`. `、`? `、`; `）截断，**绝不按字符硬截断**。

> 不做独立的句子级分词/分句——句子边界由 LLM 在抽取阶段自行判断并输出原文，省掉中英文分别处理分句逻辑的复杂度。

### 3.4 LLM 抽取

#### 3.4.1 元数据解耦提取

论文标题、作者、年份单独提取。截取论文首页（或前 2000 个字符）发送给 LLM，**仅抽取一次** `paper_title`、`paper_authors`、`paper_year` 并作为全局变量附加到该论文的所有候选句子上，避免按块重复抽取引发幻觉。

#### 3.4.2 句子抽取与分类

按文本块遍历调用 LLM，要求返回 JSON 数组。每条候选句子包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `sentence` | string | 原句 |
| `language` | string | `en` 或 `zh` |
| `function_category` | string | 六类功能分类之一（见下方） |
| `tags` | string[] | 自由标签（可选） |
| `usage_note` | string | 一句话说明适合用在什么场景 |

**功能分类体系（六类，使用后可按需调整）**

1. 研究背景引入句
2. 研究空白陈述句
3. 方法论述句
4. 结果陈述句
5. 局限性陈述句
6. 贡献总结句

#### 3.4.3 JSON 解析容错

不强依赖模型的结构化输出能力。代码层内置正则过滤机制：强制提取 LLM 返回文本中第一对 `[` 和最后一对 `]` 之间的内容，再执行 `json.loads()` 作为兜底，减少解析失败概率。

#### 3.4.4 调用策略

- 按分块分别调用，不整篇一次性丢给模型，控制每次调用的上下文范围。
- 调用失败时记录失败的分块，允许后续单独重试，不影响已成功的分块。

### 3.5 待审核队列

- 抽取完成后，候选句子写入本地 JSON 文件（不进数据库），按论文分开存放，方便按论文维度审核或重新抽取。
- 该队列为临时状态，单用户顺序访问，无需额外存储引擎。

### 3.6 交互式审核（`ppb review`）

- **中断保护**：每审核操作一条，实时更新该条目的状态标记（`status: reviewed`）。支持随时 `Ctrl+C` 退出，下次执行时从断点无缝恢复进度。
- **逐条展示**：原句、语言、功能分类、标签、来源论文（不做一次性大列表勾选，避免漏看）。
- **操作选项**：
  - `保留`：通过审核，待入库
  - `丢弃`：不进入向量库
  - `编辑`：仅可修改功能分类和标签，不修改原句文本（若原句本身有误，直接丢弃）
  - `剩余全部保留` / `剩余全部丢弃`：快速批量操作
- 只有审核通过的句子才触发 embedding 计算与入库。

### 3.7 向量化与检索

- **嵌入模型**：`BAAI/bge-m3`，通过 `sentence-transformers` 本地运行，建库与检索共用一个实例。
- **持久化存储**：ChromaDB 单机模式（`PersistentClient`），每条记录包含 embedding 向量与 metadata 字段（见第 4 节）。
- **一致性校验**：将当前使用的 embedding 模型名称与版本写入 Chroma collection 的元信息，每次启动时校验，防止建库与检索使用不一致的模型导致向量空间不匹配。
- **检索流程**：`ppb search "<查询>"` → BGE-M3 编码 → Chroma 向量检索 → 返回 Top-K 结果，展示原句、功能分类、标签、来源论文。
- 检索结果二次排序（LLM Rerank）暂不纳入 v1，列入后续规划。

## 4. 数据模型

Chroma 中每条记录的 metadata 字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | UUID 唯一标识 |
| `sentence` | string | 论文原句 |
| `language` | string | `en` / `zh` |
| `paper_title` | string | 论文标题（LLM 从首页提取） |
| `paper_authors` | string | 论文作者（LLM 从首页提取） |
| `paper_year` | string | 发表年份（LLM 从首页提取） |
| `source_file_hash` | string | 源文件内容哈希（防重复处理） |
| `function_category` | string | 六类功能分类之一 |
| `tags` | string | 自由标签，多标签逗号分隔 |
| `usage_note` | string | 使用场景说明（LLM 生成） |
| `reviewed` | bool | 是否经人工确认（入库恒为 `true`，保留以备追溯） |
| `created_at` | string | 入库时间戳（ISO 格式） |

## 5. CLI 命令设计

```
ppb extract <file>          输入 PDF 或 DOCX，自动判断格式与是否需 OCR，
                             调用 LLM 抽取候选句与论文元数据，写入待审核队列

ppb review                  交互式逐条审核候选句（支持中断恢复），
                             审核通过的写入 Chroma

ppb search "<query>"        自然语言检索，返回匹配的候选句子及元信息

ppb config                  交互式查看与修改全部配置项（LLM/OCR 密钥等），
                             无需手动编辑任何文件
```

**v1 不包含、列入后续规划的命令：**

```
ppb list --paper "<关键词>"   列出某篇论文已入库的句子
ppb stats                    查看库统计信息（总数、分类分布等）
```

## 6. 技术栈

| 模块 | 库/服务 | 选型理由 |
|------|---------|----------|
| CLI 框架 | Typer | 装饰器子命令，类型提示驱动，比 argparse 简洁，AI 辅助编码命中率高 |
| 终端交互 | questionary | 配置菜单与逐条审核交互，API 简单 |
| 终端展示 | rich | 格式化展示句子与标签，带颜色和表格 |
| PDF 解析 | PyMuPDF（fitz） | 速度快，可判断页面是否有文本层、是否含图像 |
| Word 解析 | python-docx | 标准库，读取正文与标题样式 |
| OCR | MinerU API / PaddleOCR API | 可插拔设计，处理图片型 PDF 页面 |
| LLM | 通用 OpenAI Compatible 客户端 | 支持 DeepSeek / 通义千问 / Kimi / Ollama 等 |
| Embedding | sentence-transformers + BAAI/bge-m3 | 本地开源，中英双语，无额外调用费用 |
| 向量库 | chromadb（PersistentClient） | 本机单进程运行，自带 metadata 过滤，免运维 |

## 7. 非功能需求

- **成本**：抽取阶段按分块调用 LLM API，单篇论文调用次数在个位数到十位数之间，成本量级很低。Embedding 在本地计算，无额外费用。
- **隐私**：论文正文会发送给 LLM API 做抽取；若处理未公开稿件，需评估内容外传风险。Embedding 在本地完成，不经网络。若 OCR 走云端 API，同样涉及内容外传，需注意。
- **去重**：用 `source_file_hash` 判断同一文件是否已处理，防止重复抽取。跨论文相似句不做自动去重，留待人工审核阶段肉眼判断（后续可考虑加入自动提示）。
- **可扩展性**：当前数据字段与存储结构未绑定「句子」这一种素材，未来若扩展至术语库或参考文献管理，可复用同一套抽取/审核/检索流程，新建独立的 Chroma collection 即可。

## 8. 边界情况

| 场景 | 处理方式 |
|------|----------|
| 论文同时包含文字页与扫描页 | 按页判断，两种提取方式可混用，不要求整篇统一 |
| LLM 抽取失败或返回格式不符预期 | 重试机制 + 记录失败分块，允许后续单独重跑，不影响已成功部分 |
| 同一论文重复执行 extract | 根据 `source_file_hash` 判断；若已存在，给出提示并提供 `--force` 强制重抽选项 |
| 中英文混合论文 | `language` 按整句主要语言判定，不做逐词判断 |

## 9. 版本范围

### v1.0（MVP，能跑起来）

- 首次运行 API 配置交互流 + `ppb config` 全交互式配置管理
- PDF/DOCX 解析 + OCR 兜底
- 脏数据过滤 + 多级降级分块
- 元数据解耦 + JSON 正则容错的 LLM 抽取
- 支持中断恢复的 `ppb review` 交互式审核
- BGE-M3 本地编码 + Chroma 入库 + 模型版本校验
- 自然语言纯向量召回（`ppb search`，不做二次排序）

### 后续规划（非 v1 必须）

- 检索结果的 LLM 二次排序（Rerank）
- `ppb list` / `ppb stats` 等辅助查看命令
- 跨论文相似句子的自动去重提示
- 功能分类体系的自定义扩展
